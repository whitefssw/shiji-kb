# SKILL: 每日工作日志微信群总结

## ⏰ 重要：工作日划分规则

**本系统以早上7:00为工作日分界线，而非传统的午夜0:00**

### 时间范围定义

- `YYYY-MM-DD.md` 包含的提交时间：`(YYYY-MM-DD) 07:00` 至 `(YYYY-MM-DD+1) 07:00`
- 例如：`2026-03-23.md` 包含 `2026-03-23 07:00` ~ `2026-03-24 07:00` 的所有提交

### 为什么这样设计？

1. **符合实际作息** - 凌晨0-7点的工作通常是前一天的延续
2. **方便统计** - 次日早上7点后可以完整统计前一天的工作
3. **避免切分** - 深夜加班不会被切分到两天

### 示例

| 提交时间 | 计入日志 | 说明 |
|---------|---------|------|
| 2026-03-23 08:00 | 2026-03-23.md | 白天工作算当日 |
| 2026-03-23 23:30 | 2026-03-23.md | 深夜工作算当日 |
| 2026-03-24 02:00 | 2026-03-23.md | 凌晨算前一日 |
| 2026-03-24 06:59 | 2026-03-23.md | 7点前算前一日 |
| 2026-03-24 07:00 | 2026-03-24.md | 7点起算次日 |

## 目标

根据详细的每日工作日志（markdown格式，包含多级列表和分类），生成适合微信群发送的简短平实文本，并将摘要追加到日志文件的最前面。

## 输入

用户会提供：
1. 日期（如 2026-03-17）
2. 或直接提供日志文件路径

## 工作流程

### 1. 读取日志文件

```bash
# 日志文件位置
logs/daily/YYYY-MM-DD.md
```

使用Read工具读取完整内容。

**重要**：日志文件由脚本自动生成，格式为：
```markdown
# 工作日志 YYYY-MM-DD

## 核心功能变化
...

## 技术细节
...
```

### 2. 分析日志内容

#### 2.1 检查是否有提交

先检查日志中的提交次数：
- 如果提交次数为0，跳转到 **3.2 无提交日的太史公曰**
- 如果有提交，继续正常流程

#### 2.2 提取工作内容（有提交时）

从日志中提取：
- 核心功能变化（按类别）
- 提交次数
- 关键数字（修正数、新增数等）

重点关注：
- **新功能** - 优先级最高
- **实体反思** - 核心工作
- **反思规则更新** - 方法论进展
- **Bug修复** - 重要修复
- 其他分类次之

### 3. 生成微信通知

#### 3.1 有提交日的格式

**格式要求：**

```
【史记知识库 YYYY-MM-DD】

· 核心工作1（包含关键数字）
· 核心工作2（包含关键数字）
· 核心工作3
...
· 提交N次代码
```

**具体要求：**

1. **列表格式**
   - 使用全角中点（·）作为列表符号
   - 每行一个工作项
   - 最多10项（包括"提交N次"）
   - 一般6-8项为宜

2. **纯文本格式**
   - 无markdown语法
   - 无阿拉伯数字编号
   - 无多级结构
   - 不使用emoji（除标题的【】）

3. **语言风格**
   - 平实易懂的中文
   - 避免技术黑话
   - 简化版本号（如v2.8→直接说做了什么）
   - 将技术动作转化为成果描述

4. **内容选择**
   - 提取5-9条核心工作
   - 每条尽量包含关键数字
   - 最后一项固定为"提交N次代码"

5. **优先级**
   - 带数字的成果（如"修正615处"）优先
   - 完成性工作（如"完成XX"）优先
   - 新增功能优先于维护
   - 核心模块优先于辅助工具

#### 3.2 无提交日的太史公曰

**当日无提交时，使用太史公文风书写：**

**格式要求：**

```
【史记知识库 YYYY-MM-DD】

太史公曰：是日无所书。

<32字赞文，8个四字句>
```

**赞文规范**：
- **总字数**：严格32字（**仅计汉字，不含标点**）
- **句式**：必须全部使用**四字句**（每句恰好四个汉字）
- **结构**：8个四字句 = 32字
- **文风**：参考 `labs/sima-qian-style/SKILL.md` 的太史公曰风格
- **内容**：
  - 可赞项目建设之不易
  - 可赞持续积累之重要
  - 可赞休整思考之必要
  - 可叹工程浩大需时日
- **修辞**：
  - 使用对仗、排比
  - 简短有力，点到为止
  - 避免冗长说教
  - 尽可能押韵
- **禁止**：
  - ❌ 不能使用三字句（如"久也"）
  - ❌ 不能使用五字句（如"五十七万言"）
  - ❌ 不能使用六字句或更长
  - ❌ 不重复正文内容
  - ❌ 不使用现代白话（"的""了""很"等）
- **变化性**：
  - 每天的赞文必须不同
  - 可变换主题（休整/思考/持续/工程等）
  - 可变换句式（对仗/排比/递进等）

**示例1**（休整主题，32字）：
```
大业未成，日复一日；
张弛有度，蓄力待时。
善于休整，方能长久；
厚积薄发，终有所成。
```
（8句×4字 = 32字 ✅）

**示例2**（思考主题，32字）：
```
代码暂止，思虑不息；
磨刀不误，砍柴之工。
三思后行，智者所为；
谋定后动，方克有济。
```
（8句×4字 = 32字 ✅）

**示例3**（持续主题，32字）：
```
百卷宏业，非一日成；
偶有停歇，理之常然。
持之以恒，乃成大器；
锲而不舍，金石可镂。
```
（8句×4字 = 32字 ✅）

**示例4**（工程浩大，32字）：
```
五十万言，岂能速竟？
休整思谋，长远之计。
善始善终，必有所成；
积跬致远，不可须臾。
```
（8句×4字 = 32字 ✅）

**计数方法**：
- **只计汉字，不计标点**
- 每个四字句 = 4个汉字
- 8个四字句 = 32字
- 验证方法：
  ```
  示例1验证：
  大业未成（4字）
  日复一日（4字）
  张弛有度（4字）
  蓄力待时（4字）
  善于休整（4字）
  方能长久（4字）
  厚积薄发（4字）
  终有所成（4字）

  总计：8句×4字 = 32字 ✅
  标点（不计入）：，；，。，；，。
  ```

## 实施步骤

当用户请求生成微信通知时：

1. **确认日期**
   ```
   用户：生成今天的微信通知
   或：总结2026-03-17的工作
   ```

2. **读取日志**
   ```python
   Read logs/daily/2026-03-17.md
   ```

3. **分析提取**
   - 列出所有核心变化
   - 标注优先级
   - 提取关键数字

4. **生成总结**
   - 按优先级选择3-4条
   - 简化技术术语
   - 用顿号连接
   - 加上提交次数

5. **更新日志文件**

   使用Edit工具，在日志文件的**最前面**插入微信摘要：

   ```markdown
   # 工作日志 YYYY-MM-DD

   ## 微信群通知

   ```
   【史记知识库 YYYY-MM-DD】

   · 核心工作1（包含数字）
   · 核心工作2（包含数字）
   · 核心工作3
   · 核心工作4
   · 提交N次代码
   ```

   ---

   ## 核心功能变化

   （原有内容保持不变）
   ...
   ```

   **重要**：
   - 在标题 `# 工作日志 YYYY-MM-DD` 之后立即插入
   - 使用 `## 微信群通知` 作为二级标题
   - 摘要用代码块包裹（方便复制）
   - 用 `---` 分隔线与后续内容分开
   - 不改动原有的详细内容

6. **输出确认**

   更新完成后，输出：
   ```
   ✅ 已将微信通知摘要添加到 logs/daily/YYYY-MM-DD.md

   可直接复制以下内容发送到微信群：

   【史记知识库 YYYY-MM-DD】

   核心内容1、核心内容2、核心内容3。提交N次。
   ```

## 质量检查

生成后检查：
- [ ] 是否纯文本（无markdown语法）
- [ ] 是否有编号或层级结构（应该没有）
- [ ] 是否超过150字（应该≤150）
- [ ] 是否包含关键数字
- [ ] 是否易于非技术人员理解
- [ ] 是否适合直接复制到微信群
